# 휴대폰 기종별 결정 문서 사이트 — 설계 문서 v2

> 콘셉트 전환: "휴대폰의 모든 것(블로그)" → **"기종별 결정 문서 DB"**
> 사이트가 답하는 단 하나의 질문: **"이 폰, 사도 되나 / 계속 써도 되나?"**
>
> v1 대비 핵심 변경
> - `/phones/[model]` 곁다리 허브 → **사이트의 본체**
> - 콘텐츠(글) 중심 → **구조화된 데이터(표) 중심**. 글은 데이터를 보조
> - 작업량 무한(모든 문서) → **유한(기종 × 문서 5종)**
> - 다나와·모요·GSMArena와 정면승부 회피 — 그들이 안 하는 문서만 만든다

## 1. 서비스 개요

| 항목 | 내용 |
|---|---|
| 콘셉트 | 기종별 "구매·유지 결정"에 필요한 문서 5종을 표준화해 제공 |
| 핵심 차별화 | ①업데이트 종료 캘린더 ②공식 수리비 DB ③고질병 문서 ④구매 루트 요약 ⑤잔존가치 곡선 — **한국에서 아무도 기종 단위로 정리하지 않는 데이터** |
| 경쟁 회피 | 스펙 상세 → GSMArena/제조사에 양보. 실시간 최저가 → 다나와 아웃링크. 요금제 조합 → 모요 영역, 요약+아웃링크만 |
| 수익 모델 | 쿠팡 파트너스(자급제 아웃링크) → 애드센스 → 알뜰폰 개통 CPA. 장기: 잔존가치 리포트 = 백링크·브랜드 자산 |

| AI 제로클릭 대응 | 프로즈가 아니라 **기준일 있는 표·계산·시계열** — AI가 요약해도 "최신 원본"을 확인하러 오게 만드는 구조 |

- 『트렌드 코리아 2026』 "프라이스 디코딩" — 가격 뒤 구조를 따지는 소비 트렌드와 정합
- 문제해결(how-to) 콘텐츠는 AI Overview 제로클릭에 가장 취약 → 데이터 표 중심으로 전환

## 2. 핵심 제품: 기종 문서 5종

모든 기종은 **동일한 구조의 문서 세트**를 갖는다. 기준: "결정에 영향을 주는가?"

| # | 문서 | 답하는 질문 | 데이터 성격 | 갱신 주기 |
|---|---|---|---|---|
| D1 | 업데이트 종료 캘린더 | 몇 년 더 쓸 수 있나? | OS/보안 지원 종료일 | 반기 검증 + 정책 변경 시 |
| D2 | 공식 수리비 | 고장 나면 얼마 드나? | 화면·배터리·후면 등 부품별 공식가 | 분기 |
| D3 | 고질병·알려진 이슈 | 이 기종 특유의 문제는? | 이슈 목록 + 해결법 + 리콜/무상수리 여부 | 수시 (MDX) |
| D4 | 구매 루트 | 지금 어디서 사는 게 최선? | 자급제 아웃링크·공시지원금 요약·알뜰폰 노트 | 월 1회 |
| D5 | 잔존가치 | 3년 뒤 얼마 남나? | 중고 시세 시계열 (출시가 대비 %) | 월 1회 기록 |

**철칙**
- 모든 데이터 셀에 `기준일(asOf)`과 `출처(source)` 표기. 오래된 시세·가격은 신뢰를 즉사시킨다 — 갱신 못 한 데이터는 흐리게 처리 + "확인 필요" 배지
- D4는 가격 비교 엔진을 만들지 않는다. **요약 + 아웃링크(제휴)**로만 처리 — 다나와·모요와 싸우지 않고 수수료만 가져간다
- D3만 자유 서술(MDX), 나머지는 전부 구조화 데이터 → 렌더링

## 3. 정보 구조 (IA) / 사이트맵

```
/                              홈: 검색창 + 최근 갱신 + 이번 달 지원종료 임박 기종
├── /phones                    기종 인덱스 (제조사·연도·가격대 필터)
│   └── /phones/[model]        ★ 기종 허브 = 본체 (문서 5종 요약 + 결정 요약 카드)
│       ├── /updates               D1 전문 (검색 랜딩: "○○ 언제까지 업데이트")
│       ├── /repair                D2 전문 ("○○ 액정 수리비")
│       ├── /issues                D3 전문 ("○○ 발열/결함")
│       ├── /buy                   D4 전문 ("○○ 싸게 사는 법")
│       └── /resale                D5 전문 ("○○ 중고 시세")
├── /compare/[a]-vs-[b]        기종 비교 — 스펙 나열이 아니라 "지원종료·수리비·잔존가치" 비교로 차별화
├── /calendar                  전 기종 업데이트 종료 캘린더 (모아보기 = 링크베이트 겸 도구)
├── /repair-cost               전 기종 수리비 비교 표
├── /report/[slug]             분기 잔존가치 리포트 (언론 인용용, 보도자료 겸)
├── /guide/[slug]              기종 무관 공통 가이드 (침수 대처, 중고 거래 체크리스트 등 — 소수만)
├── /sitemap.xml, /robots.txt, /rss.xml
```

- 규모: 30기종 × 6페이지(허브+5) = **180페이지** + 비교/캘린더/리포트. 유한하고 계획 가능
- 검색 랜딩은 하위 문서 페이지(`/phones/galaxy-s24/repair`)가 담당, 허브는 내부 회유·체류 담당
- 비교 페이지는 canonical 한 방향 + **역방향 URL 301 리다이렉트**
- v1의 `/review` `/tips` 삭제. `/fix`는 D3(기종 이슈)와 `/guide`(공통)로 흡수

## 4. 데이터 모델

기종당 JSON 1파일 (`data/phones/galaxy-s24.json`) + Zod 스키마 검증. D3 본문만 MDX.

```ts
interface Phone {
  slug: string                  // galaxy-s24
  name: string                  // 갤럭시 S24
  brand: 'samsung' | 'apple' | ...
  releaseDate: string           // 2024-01-31
  releasePriceKRW: number       // 출시가 (잔존가치 계산 기준)
  specSummary: SpecSummary      // 최소한만: 칩셋/램/화면/배터리 — 상세는 외부 링크

  eol: {                        // D1
    lastMajorOs: string         // 예: One UI 9 / Android 17
    osEndDate: string | null    // 메이저 업데이트 종료 (추정 시 estimated: true)
    securityEndDate: string | null
    estimated: boolean
    source: SourceRef
  }

  repairCosts: Array<{          // D2
    part: 'display' | 'battery' | 'back-glass' | 'camera' | 'charging-port'
    officialKRW: number
    withCareKRW?: number        // 케어플러스/애플케어 적용가
    source: SourceRef
  }>

  issues: Array<{               // D3 (메타만 JSON, 본문은 MDX)
    id: string
    title: string               // 예: "GPS 튐 현상"
    severity: 'info' | 'common' | 'critical'
    status: 'open' | 'patched' | 'recall' | 'free-repair'
    mdxSlug: string
  }>

  buyRoutes: {                  // D4
    coupangUrl?: string         // 파트너스 링크
    danawaUrl?: string          // 비제휴 아웃링크 (신뢰용)
    subsidyNote: string         // 공시지원금 요약 (스마트초이스 기준)
    mvnoNote: string            // 알뜰폰 조합 한 줄 요약 + 모요 아웃링크
    asOf: string
  }

  resale: Array<{               // D5 시계열
    date: string                // 월 1회
    priceKRW: number            // 중고 시세 중앙값 (상태 A급 기준)
    source: 'bunjang-median' | ...
  }>
}

interface SourceRef { label: string; url: string; asOf: string }
```

**파생 지표 (자동 계산)**
- `잔존가치율` = 최신 resale / releasePriceKRW
- `남은 수명` = securityEndDate − today → 허브의 "결정 요약 카드"와 `/calendar`에 사용
- `수리비 부담률` = display 수리비 / 최신 중고 시세 (수리 vs 교체 판단 지표 — 독자적 지표라 인용가치 있음)

## 5. 데이터 소스 및 수집 방법

| 데이터 | 소스 | 수집 방법 | 비용/난이도 |
|---|---|---|---|
| D1 지원 종료 | 삼성 보안 업데이트 정책 페이지, Apple 보안 지원 문서, endoflife.date | 수동 정리 + 반기 검증. 종료일 미공표 기종은 정책(연수)으로 추정 후 `estimated` 표기 | 낮음 |
| D2 수리비 | 삼성전자서비스 부품가격 조회, Apple 수리 비용 페이지 | 수동 분기 갱신 (30기종 × 3~5부품 = 감당 가능) | 낮음 |
| D3 고질병 | 커뮤니티·뉴스·제조사 공지 리서치 | 직접 작성 (MDX). 리콜·무상수리는 공식 공지 링크 필수 | 중간 — 유일한 "취재" 영역 |
| D4 공시지원금 | 스마트초이스 (방통위) | 요약 문장 월 1회 수동 갱신. 상세는 아웃링크 | 낮음 |
| D4 자급제가 | 쿠팡 파트너스 / 다나와 | 링크만. **실시간 가격 표시 안 함** (크롤링 부담·정확성 리스크 회피) | 낮음 |
| D5 중고 시세 | 번개장터 시세 검색 (중앙값 수기 기록) | Phase 1~2: 월 1회 수동 기록 (30기종 × 월 1회 = 2시간). Phase 3: 자동화 검토 | 낮음→중간 |

> 원칙: **자동화되기 전까지는 수동 주기를 지킬 수 있는 규모만 다룬다.** 기종 수 상한 40. 갱신 못 지킬 데이터는 싣지 않는 게 낫다.

## 7. SEO 설계

### 7.1 문서 타입별 타깃 쿼리·제목 공식

| 문서 | 타깃 쿼리 예 | 제목 공식 |
|---|---|---|
| D1 | "갤럭시 S23 언제까지 업데이트" | `[기종] 업데이트 종료일 — OS·보안지원 (기준일)` |
| D2 | "아이폰 15 액정 수리비" | `[기종] 수리비 총정리: 화면·배터리 공식가 (2026)` |
| D3 | "갤럭시 S24 발열" | `[기종] 알려진 문제 [N]가지와 해결법` |
| D4 | "갤럭시 S25 싸게 사는 법" | `[기종] 최저 구매 루트: 자급제 vs 지원금 (월 갱신)` |
| D5 | "아이폰 14 중고 시세" | `[기종] 중고 시세 추이 — 잔존가치 [N]% (월 갱신)` |

### 7.2 구조화 데이터

| 페이지 | 스키마 |
|---|---|
| 기종 허브 | `Product` + `BreadcrumbList` |
| D1~D5 | `Article` + `FAQPage` (frontmatter faq 자동 생성 — v1 방식 유지) |
| /calendar, /repair-cost | `ItemList` |
| 전체 | `WebSite` + `Organization` |

### 7.3 성능·기타

- 표 중심 페이지 → featured snippet 최적화 (표 마크업은 시맨틱 `<table>`)
- `next/image`, `next/font` 셀프호스팅 — v1 유지
- **애드센스 슬롯은 도입 전부터 `min-height` 예약** — CLS < 0.1은 나중에 광고 붙이면 못 지킨다
- 내부 링크: 허브↔문서 5종 상호, 시리즈 간(S23→S24), 비교 페이지가 두 허브 연결
- 갱신일(`updatedAt`) 노출 — 월 갱신 데이터는 그 자체가 신선도 신호
